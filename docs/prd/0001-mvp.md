# PRD 0001 — Cybus MVP: live buses + stop arrivals

> Status: ready-for-agent (label could not be applied — no issue tracker configured; this PRD lives in-repo instead). Vocabulary follows [CONTEXT.md](../../CONTEXT.md); respects [ADR-0001](../adr/0001-free-tier-first-realtime-architecture.md) and [ADR-0002](../adr/0002-ios-only-native-swift.md).

## Problem Statement

People who rely on buses in Cyprus — tourists, teenagers, and anyone who doesn't drive — have no easy, trustworthy way to know where a bus is right now or when it will reach their Stop. Cyprus's buses aren't on Google Maps, and the official real-time data, while public, is raw GTFS / GTFS-realtime that an ordinary rider can't use. The result is uncertainty at the stop: "Is a bus even coming? When? Did I just miss it?"

## Solution

A free, native iOS app that shows Cyprus's buses live on a map and tells you when they'll reach a Stop. The user opens the app to a map of moving **Vehicle** markers; they tap any **Stop** to see a time-ordered board of upcoming **Arrivals**, each clearly marked as a live prediction or a scheduled fallback. Tapping an Arrival highlights and follows that **Vehicle** on the map. No account, no setup, no cost to the user — open it and see what's coming.

## User Stories

1. As a rider, I want to see buses moving on a map of Cyprus, so that I can tell at a glance where service is active near me.
2. As a rider, I want the map to open quickly to a useful view, so that I get value before granting any permissions.
3. As a rider, I want to tap a "locate me" control to center the map on my position, so that I can see the buses around me without scrolling.
4. As a privacy-conscious rider, I want the app to ask for location only when I tap that control, so that I'm not prompted before I see any value.
5. As a returning rider, I want the map to reopen on the area I last looked at, so that I don't have to navigate back to my city each time.
6. As a first-time user, I want the map to open on a sensible default view of Cyprus, so that the app is useful before it knows my location.
7. As a rider, I want individual Stops to appear as I zoom in, so that the map isn't an unreadable wall of pins when I'm zoomed out.
8. As a rider, I want to tap a Stop to see what's coming, so that I can decide whether to wait.
9. As a rider, I want upcoming Arrivals listed soonest-first across all Routes, so that I immediately see the next bus out of that Stop.
10. As a rider, I want each Arrival to show its Route and destination, so that I know whether it's the bus I want.
11. As a rider, I want live Arrivals shown as a countdown with a clear "live" indicator, so that I trust the time is real.
12. As a rider, I want scheduled-only Arrivals shown as a clock time and clearly labelled "scheduled", so that I understand that time is not a live prediction.
13. As a rider, I want imminent Arrivals shown in minutes and far-off ones shown as a clock time, so that the times are easy to read at the relevant timescale.
14. As a rider, I want to tap a live Arrival to highlight and follow its Vehicle on the map, so that I can see exactly where my bus is.
15. As a rider, I want tapping a live Arrival to draw that Vehicle's Route line, so that I understand the path it will take to me.
16. As a rider, I want tapping a scheduled Arrival (no Vehicle yet) to draw its Route line to the Stop, so that I still get spatial context.
17. As a rider, I want the stop detail to appear as a sheet over the map, so that I keep my place on the map while reading Arrivals.
18. As a rider, I want the live board to refresh on its own while I watch, so that the countdowns stay current without me reloading.
19. As a rider, I want Route lines drawn only when I select something, so that the default map stays clean and readable.
20. As a rider, I want Vehicle markers distinguishable by Route, so that I can tell different lines apart on the map.
21. As a rider in any Cypriot city, I want coverage of all Providers (EMEL, OSYPA, OSEA, NPT, LPT, Intercity, Pame Express), so that the app works wherever I am on the island.
22. As a rider, I want the app to keep working when a Provider's real-time feed is down, by falling back to scheduled times, so that I still get information during outages.
23. As a rider, I want the app to behave sanely at night when almost no Vehicles are running, so that an empty map reads as "no service now" rather than "broken app".
24. As a rider on cellular, I want the app to be light on mobile data, so that watching the map doesn't cost me much.
25. As a rider, I want the app to stop polling when it's in the background, so that it doesn't drain my battery or data when I'm not looking.
26. As a rider, I want static Route/Stop data available without a fresh download every launch, so that the app is fast and works on a weak connection.
27. As a rider, I want the app to download updated Route/Stop data when it changes, so that I'm not stuck with stale routes after a network change.
28. As a non-English speaker, I want Stop and Route names shown in the names the feed provides, so that they match the signage I see.
29. As the maintainer, I want the static GTFS rebuilt and republished on a schedule, so that the app's reference data stays current without manual work.
30. As the maintainer, I want a single real-time entry point shared by all devices, so that one upstream fetch serves everyone and stays within free tiers.
31. As the maintainer, I want the app to verify the integrity of the data file it downloads, so that a corrupt or partial download can't crash users.
32. As the maintainer, I want malformed or hostile feed input rejected before it reaches users, so that a bad feed day degrades gracefully instead of breaking the app.
33. As the maintainer, I want to run the whole thing within free tiers, so that the app costs nothing beyond the Apple Developer fee.

## Implementation Decisions

**Platform & rendering.** Native iOS, SwiftUI + Apple MapKit (free maps, no per-load billing) — see ADR-0002. No Android, no cross-platform layer for the MVP.

**Two backend pieces, both free-tier** (see [data-pipeline.md](../data-pipeline.md)):
- A **scheduled build job** (default: a GitHub Action on a cron) downloads the seven per-Provider GTFS zips, validates them, merges them into one compact SQLite containing only what the app needs (stops, routes, trips, shapes), computes a SHA-256, and publishes the SQLite plus a small version manifest to free static hosting.
- A **thin edge proxy** (default: a Cloudflare Worker) fetches the combined GTFS-realtime protobuf once per ~30–60s, edge-caches the snapshot, and serves it to all devices over HTTPS. The proxy is mandatory because the upstream is plain HTTP on an IP:port that iOS App Transport Security would block. Per ADR-0001.

**On-device data layer (the primary seam).** The app downloads the static SQLite once and refreshes it only when the manifest version changes (verifying the SHA-256 first). It polls the proxy for the real-time protobuf (~30–60s, foreground only), decodes it on-device, and performs all joins locally:
- VehiclePositions → **Vehicle** markers, coloured by **Route** (joined via the static SQLite).
- TripUpdates → **Arrival** list for a given **Stop**, each tagged live (from the feed) or scheduled (fallback from static GTFS when the **Trip** is silent), ordered soonest-first.
- Arrival → **Vehicle** correlation for tap-to-track, via `trip_id`; when no live Vehicle resolves, fall back to drawing the Route line only.

This layer is expressed as a pure transformation so it can be tested without network or UI. The decision-bearing shape (from planning, trimmed):

```
// inputs: an opened static snapshot + a decoded GTFS-RT FeedMessage + a query
buildVehicles(static, feed) -> [Vehicle]            // positions joined to routes
buildArrivals(static, feed, stopId, now) -> [Arrival] // soonest-first, each .live or .scheduled
resolveVehicle(feed, arrival) -> Vehicle?            // trip_id correlation for tap-to-track
```

**Data sources** (verified — see [data-sources.md](../data-sources.md)):
- Real-time: combined GTFS-RT protobuf at the CyNAP web service; no key. Source cadence is "up to 1 minute", which sets the proxy cache window.
- Static: seven per-Provider zips on `motionbuscard.org.cy`; download requires the `&rel=True` query param. CC BY 4.0, no key.

**Map behaviour.** Default view = last-viewed region (island fit on first launch). Location requested in-context via a "locate me" control. Vehicle markers always shown; Stops appear above a zoom threshold; Route lines drawn only on selection.

**Stop detail.** A draggable bottom sheet over the map; a time-ordered Arrival board; live = countdown + live indicator, scheduled = muted clock time + "scheduled" label; near-term in minutes, far-off as clock time; self-refreshing while open.

**No user state.** No accounts, no favourites, no user database; nothing stored server-side about users. The only persisted local state is UI state (last-viewed region) and the cached static snapshot.

**Security posture** (proportionate to public, low-stakes data — see [security.md](../security.md)): validate at our two controlled pieces; the app trusts only HTTPS-served, hash-verified artifacts and range-checks values (e.g. coordinates) before rendering.

## Testing Decisions

**What makes a good test here:** assert external behaviour at a seam, not implementation details. Feed fixtures in, assert on the domain values out. No assertions about internal structures, private helpers, or MapKit internals. Tests must not hit the network — fixtures stand in for the static SQLite and the GTFS-RT feed.

**Primary seam — the on-device domain/data layer.** This is where correctness lives and gets the most coverage. Tests feed a fixture static snapshot plus a fixture decoded FeedMessage and assert on:
- Vehicles produced, with correct Route association.
- Arrival board for a Stop: ordering (soonest-first), live-vs-scheduled tagging, the scheduled fallback when a Trip has no real-time data, and minutes-vs-clock formatting boundaries.
- Arrival→Vehicle correlation: resolves the right Vehicle when present; falls back cleanly (Route line only) when `trip_id` has no matching Vehicle.
- Edge cases: night-time empty feed (no Vehicles), out-of-range coordinates dropped/clamped, a Trip present in static but absent from real-time.

**Secondary seam — the static build pipeline.** Feed fixture GTFS zips, assert the merged SQLite has the expected schema and known rows (e.g. Limassol's 244 Routes / 1,407 Stops), and that fetch hardening rejects a non-zip response (the `200`-HTML-error-page case) and enforces the magic-byte/size checks.

**Thin coverage — the edge proxy.** A handful of handler tests with a stubbed upstream: returns the cached snapshot, serves over the expected content type, and honours the cache window. Little logic, so little testing.

**Prior art.** None yet — this is a greenfield repo. These become the reference patterns for future tests: fixture-driven, seam-level, network-free.

## Out of Scope

- Android or any cross-platform build (ADR-0002).
- Trip planning / A→B routing, transfers, or journey suggestions.
- Accounts, login, favourites, saved Stops/Routes, or cross-device sync.
- Push notifications / arrival alerts.
- Server-side joins, a full ingest backend, or any always-on server (ADR-0001).
- Cryptographic signing of the static data file (SHA-256 manifest only — see security.md).
- Ads, in-app purchases, or any monetisation.
- Service alerts beyond what's needed for the live/scheduled distinction.
- Cert pinning and defence against a fully compromised static host (explicit security non-goals).

## Further Notes

- Verified at planning time but worth re-checking against a live **daytime** feed: the `trip_id`↔Vehicle correlation for tap-to-track (some feeds omit Vehicle info on TripUpdates), and which Providers actually emit real-time vs schedule-only. The feed sampled during planning was at ~02:30 Cyprus time with almost no Vehicles running.
